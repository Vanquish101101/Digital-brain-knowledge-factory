### Этап 2 — Docker-инфраструктура
`docker-compose.yml`: Qdrant + Postgres/pgvector + MinIO + Redis, профиль **LIGHT**
(машина пользователя — 16GB RAM, ~7GB свободно; FULL-профиль с OpenSearch+Neo4j на JVM
не влезет). Подключён внешний диск `E:` — 1.7TB свободно (на нём же лежит **посторонняя**
папка «Игры пс4», НИКОГДА её не трогать/не сканировать).

### Этап 3 — kf.py: полный пайплайн приёма и поиска знаний
Написан через **строгий TDD** (RED-GREEN на каждый модуль, часть тестов — интеграционные,
против реально поднятых контейнеров, без моков). Модули: `scope` (фильтр что индексировать),
`chunking` (разбивка текста), `hashing` (sha256 для идемпотентности), `extract` (pdf/docx/
md/txt), `config`, `embeddings` (fastembed, локально, модель
`paraphrase-multilingual-MiniLM-L12-v2`, 384d), `store/{qdrant,postgres,minio}_store`,
`ingest` (оркестрация), `llm` (OpenRouter), `api` (общая бизнес-логика), `cli` (kf.py),
`mcp_server` (FastMCP). Сейчас **51 тест**, все зелёные.

**Реальные баги, найденные и починенные на боевом прогоне** (важно помнить, если что-то
подобное всплывёт снова):
1. **Windows-путь ломал ключи объектов MinIO.** `path.relative_to()` на Windows даёт `\`,
   а S3/MinIO ждёт `/`. Фикс: `.as_posix()` вместо `str()`.
2. **Таймаут Qdrant на большом файле.** Заливка тысяч чанков одним HTTP-запросом (файл
   закладок браузера, 4.2MB, ~19000 ссылок) упиралась в таймаут. Фикс: батчинг по 64 точки
   в `upsert_chunks`. Этот файл закладок теперь вообще исключён из индексации
   (`EXCLUDED_FILENAMES` в `kf/scope.py`) — низкая ценность для поиска.
3. **Клиентский таймаут ≠ серверный сбой.** Тот самый упавший (до фикса батчинга) запрос
   на самом деле **успел записать** ~3000 «висящих» чанков на сервере Qdrant, хотя клиент
   получил ошибку и ничего не записал в Postgres (значит для `kf.py` файл выглядел как
   «не проиндексирован», а в Qdrant дубликаты уже лежали). Почищено функцией
   `delete_by_path`. **Урок: если Qdrant/любой сетевой сервис кидает клиентский таймаут,
   не считай автоматически, что операция не применилась на сервере — стоит проверить.**

### Этап 6 — переименования (дважды)
1. `knowledge-factory/` → `Фабрика знаний (Цифровой мозг)/` (2026-07-11)
2. `Структура хранилища/` → `Хранилище входных данных/`,

   `E:\knowledge-factory\` → `E:\Фабрика знаний (knowledge-factory)\` (2026-07-12)

**Ключевые технические уроки, которые почти наверняка всплывут снова, если будут ещё
переименования/переносы:**
- Запущенный MCP-сервер (процесс `python -m kf.mcp_server`) держит директорию открытой —
  простой `Rename-Item` падает с «in use». Порядок действий: `claude mcp remove` (иначе
  Claude Code переподключает его в фоне быстрее, чем успеваешь переименовать) → убить
  оставшиеся процессы (`Get-CimInstance Win32_Process | Where CommandLine -like
  "*kf.mcp_server*"` → `Stop-Process`) → и только потом переименовывать/перемещать.
- Если `Rename-Item`/`Move-Item` всё равно падает «in use» — использовать `robocopy /E
  /MOVE` (сразу с `/XD ".venv"`, чтобы не тащить venv, который всё равно надо пересоздавать).
- `.venv` **никогда не копировать** при переносе — в нём зашиты абсолютные пути. Всегда
  `uv sync` заново на новом месте.
- **Docker Compose берёт имя "проекта" из имени текущей папки.** Если у именованного тома
  (`volumes: kf_postgres_data:`) нет явного `name:`, при переименовании папки Docker
  попытается создать **новый пустой** том вместо переиспользования старого с данными —
  реальный риск незаметной потери данных. Уже исправлено: в `docker-compose.yml` у тома
  Postgres прописано `name: knowledge-factory_kf_postgres_data` явно, не зависит от папки.
- `docker compose down` у этого пользователя **периодически молча ничего не делает**
  (exit code 0, но контейнеры остаются Up). Работает через раз. Если это увидишь — не жди,
  сразу используй `docker stop <контейнер>` напрямую на каждый контейнер.
- **Docker Desktop у пользователя на Hyper-V, не WSL2** (старая версия Windows). Менее
  стабилен с проброшенными портами, особенно после перезапуска хоста/системы — иногда рвутся
  уже установленные соединения (`Software caused connection abort`), помогает
  `docker restart <контейнер>`. Изредка сам демон коротко обрывает соединение
  (`error during connect ... EOF`) — но не значит, что завис насмерть, `docker ps` может
  тут же снова ответить нормально. **Правило от пользователя: если Docker подряд долго не
  отвечает — не пытаться чинить бесконечно самому, сказать ему, чтобы перезапустил Docker
  Desktop вручную.**
- После сбоя `docker compose up` контейнеры иногда создаются с кривыми хэш-префиксными
  именами (`8af435a60040_kf-qdrant` вместо `kf-qdrant`) — это артефакт гонки при пересоздании,
  не страшно, просто `docker rm -f` и поднять ещё раз.
- Смешение кириллицы/латиницы/пробелов/скобок в путях **само по себе багов не создаёт** —
  Python, Docker, git, MinIO, Qdrant, PowerShell 5.1+ современные, все UTF-8-safe. Реальная
  причина всех багов, что ловили — Windows-специфика (разделители путей, права NTFS,
  блокировки процессов), не язык имён.
