## Ключевые решения

| Вопрос | Решение | Почему |
|---|---|---|
| Профиль стека | **LIGHT** (Qdrant + Postgres/pgvector + MinIO + Redis + Kuzu, без OpenSearch/Neo4j) | RAM всего 16GB, доступно ~7GB — FULL-профиль (OpenSearch+Neo4j на JVM) не влезет |
| LLM для графа/синтеза | **OpenRouter** (через LiteLLM) | Проще при 16GB RAM, не ест локальную память; эмбеддинги всё равно локальные (fastembed) |

| Писать ли `kf.py` (пайплайн приёма/эмбеддинга + Knowledge MCP) сейчас | **Нет, отдельной сессией** | Это самостоятельный кусок кода (парсинг, чанкинг, Qdrant/Postgres клиенты, MCP-сервер) — логично писать и тестировать после того, как инфраструктура (`docker compose up`) подтверждённо работает |

Отклонения от плана, оба вынужденные:
- **Redis на порту 6380**, не 6379 — порт 6379 занят чужим контейнером `marketing-agency-redis`
  (другой проект пользователя, работает параллельно, не трогали).
- **Postgres на именованном томе Docker, не на `E:`** — Windows-ограничение: bind-mount NTFS
  не может отдать контейнеру строгие Unix-права, которые Postgres требует для своей data-папки
  (проверено: и корень точки монтирования, и подпапка `PGDATA` — обе попытки упали с
  `FATAL: data directory has invalid permissions`). Том физически живёт внутри диска Docker
  Desktop (обычно на `C:`), НЕ на внешнем диске. Для Digital Brain это некритично — в Postgres
  тут только метаданные/текст чанков, не сырые файлы и не векторы (то и другое — на `E:`,
  через Qdrant и MinIO), так что рост на `C:` должен быть небольшим. Если захочется строгого
  соответствия правилу "данные не на системном диске" — можно перенести весь диск образов
  Docker Desktop на `E:` через Settings → Resources → Advanced → Disk image location
  (разовая настройка через GUI, после неё все именованные тома автоматически окажутся на `E:`).

## Приём картинок и видео в kf.py (2026-07-12)

По запросу «реализуй расширение kf.py для картинок и видео». Полный цикл: brainstorming →
спека (`docs/superpowers/specs/2026-07-12-kf-image-video-ingest-design.md`) → план
(`docs/superpowers/plans/2026-07-12-kf-image-video-ingest.md`) → реализация 9 задач через
subagent-driven-development, напрямую в `main` (без отдельной ветки, по решению пользователя —
"не плодить множество версий"), каждая задача с отдельным ревью, плюс финальное ревью всей ветки.

**Архитектура — гибрид, локально в приоритете, облако как fallback:**
- Картинки: локальный OCR (Tesseract, `kf/ocr.py`, языки `rus+eng`) в приоритете; если
  текста мало (< `IMAGE_CAPTION_THRESHOLD_CHARS`, по умолчанию 20 символов) — добавляется
  подпись через vision-модель OpenRouter (`kf/vision_caption.py`, `google/gemini-2.5-flash`).
- Видео: аудио-транскрипт целиком (`kf/transcribe.py`, faster-whisper, модель `small`,
  автоопределение языка) + сэмплированные кадры каждые `VIDEO_FRAME_INTERVAL_SECONDS`
  (по умолчанию 15 сек, `kf/video.py` через ffmpeg) — каждый кадр проходит через тот же
  пайплайн, что и картинка.
- Всё полностью автоматически через `kf.py ingest`, без ручных шагов на файл.
- Новые системные зависимости (устанавливаются вручную, не через код) — Tesseract OCR
  (+ языковой пакет `rus`) и ffmpeg, см. `ПЕРЕД-СТАРТОМ.md`. На этой машине уже установлены.

**Найденные и исправленные дефекты в процессе (не попавшие в финальный код):**
1. Коллизия временных путей в `extract_audio()` (одинаковое имя видео из разных папок
   вокруг могло перезаписать чужой временный wav) — исправлено через `tempfile.mkstemp`.
2. Захардкоженный `language="en"` в `transcribe_audio()` (implementer сломал автоопределение
   языка, пытаясь победить нестабильный тест) — исправлено, добавлен опциональный параметр
   `language=None` по умолчанию (авто), тест передаёт `"en"` только из своего вызова.
3. Финальное ревью всей ветки (opus) нашло утечку: временные wav/кадры видео никогда не
   удалялись после обработки, хотя спека это явно требовала — исправлено отдельным коммитом
   после ревью (`sample_frames()` теперь возвращает и директорию с кадрами, `_extract_video()`
   удаляет wav и директорию в `finally`, включая случай ошибки на кадре).

**Известное, оставленное как есть:** несколько Minor-находок ревью (нет end-to-end теста с
реальным видео — закрыто ручной проверкой ниже; `test_transcribe.py` обнуляет кэш модели через
`tmp_path`; тайм-коды кадров приблизительные; один сбойный vision-вызов роняет подписи всех
кадров видео) — не исправлены, не блокируют.

Проверено: полный набор тестов — **65/65 пройдено** (со поднятым Docker-стеком). Все коммиты
запушены в GitHub (`origin/main`, `9b1e77c`).

### Лимит кадров/стоимости vision-API на видео (V1.4.15)

Закрыт Important #2 финального ревью (был оставлен как соображение, не дефект): без лимита
часовое видео при интервале 15 сек дало бы 240 вызовов vision-модели на один файл — риск
неконтролируемых расходов на OpenRouter. Добавлена настройка `MAX_VIDEO_FRAMES` (по умолчанию
**20** — покрывает первые ~5 минут видео при интервале 15 сек), которая ограничивает верхний
предел кадров, извлекаемых из видео за один `ingest`, независимо от его длительности.

Реализация — в `kf/video.py::sample_frames()`, через нативную опцию ffmpeg `-frames:v N`
(накладывается поверх фильтра `fps=1/N`, не меняет поведение при коротких видео — фолбэк
"хотя бы 1 кадр" из V1.4.13 работает как прежде). TDD: `test_sample_frames_respects_max_frames_cap`
(RED → GREEN, реальный ffmpeg, без моков). `_extract_video()` прокидывает `settings.max_video_frames`
в `sample_frames()` — отдельный тест `test_extract_video_passes_max_frames_cap_to_sample_frames`
проверяет проброс значения.

Проверено сквозным реальным прогоном `kf.py ingest` (временно `VIDEO_FRAME_INTERVAL_SECONDS=1`,
`MAX_VIDEO_FRAMES=2` для теста, не в `.env`): 6-секундное видео без лимита дало бы ~6 кадров,
с лимитом — ровно 2, каждый прошёл через реальный vision-fallback (OpenRouter). Полный набор
тестов: **68/68**. `.env.example` документирует новую переменную.
