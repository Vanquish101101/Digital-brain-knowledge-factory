## Ключевые решения

| Вопрос | Решение | Почему |
|---|---|---|
| Профиль стека | **LIGHT** (Qdrant + Postgres/pgvector + MinIO + Redis + Kuzu, без OpenSearch/Neo4j) | RAM всего 16GB, доступно ~7GB — FULL-профиль (OpenSearch+Neo4j на JVM) не влезет |
| LLM для графа/синтеза | **OpenRouter** (через LiteLLM) | Проще при 16GB RAM, не ест локальную память; эмбеддинги всё равно локальные (fastembed) |
| Куда класть тома данных | **Решено** — `E:\Фабрика знаний (knowledge-factory)\data` (внешний диск, 1.7TB свободно) | Диск подключён 2026-07-11. Рядом на `E:` лежит папка «Игры пс4» (144.5GB) — не относится к проекту, **не сканировать, не трогать, не удалять** |
| Писать ли `kf.py` (пайплайн приёма/эмбеддинга + Knowledge MCP) сейчас | **Нет, отдельной сессией** | Это самостоятельный кусок кода (парсинг, чанкинг, Qdrant/Postgres клиенты, MCP-сервер) — логично писать и тестировать после того, как инфраструктура (`docker compose up`) подтверждённо работает |

## Стек поднят (2026-07-11)

| Контейнер | Порт (хост) | Данные | Статус |
|---|---|---|---|
| kf-qdrant | 6333/6334 | `E:\Фабрика знаний (knowledge-factory)\data\qdrant` | healthy |
| kf-postgres | 5432 | именованный том Docker `kf_postgres_data` (не на E:, см. ниже) | healthy |
| kf-minio | 9000/9001 | `E:\Фабрика знаний (knowledge-factory)\data\minio` | healthy |
| kf-redis | **6380** (не 6379!) | `E:\Фабрика знаний (knowledge-factory)\data\redis` | healthy |

Отклонения от плана, оба вынужденные:
- **Redis на порту 6380**, не 6379 — порт 6379 занят чужим контейнером `marketing-agency-redis`
  (другой проект пользователя, работает параллельно, не трогали).
- **Postgres на именованном томе Docker, не на `E:`** — Windows-ограничение: bind-mount NTFS
  не может отдать контейнеру строгие Unix-права, которые Postgres требует для своей data-папки
  (проверено: и корень точки монтирования, и подпапка `PGDATA` — обе попытки упали с
  `FATAL: data directory has invalid permissions`). Том физически живёт внутри диска Docker
  Desktop (обычно на `C:`), НЕ на внешнем диске. Для Digital Brain это некритично — в Postgres
  тут только метаданные/текст чанков, не сырые файлы и не векторы (то и другое — на `E:`,
  через Qdrant и MinIO), так что рост на `C:` должен быть небольшим. Если захочется строгого
  соответствия правилу "данные не на системном диске" — можно перенести весь диск образов
  Docker Desktop на `E:` через Settings → Resources → Advanced → Disk image location
  (разовая настройка через GUI, после неё все именованные тома автоматически окажутся на `E:`).

## Переименование папки (2026-07-11)

`knowledge-factory/` → `Фабрика знаний (Цифровой мозг)/`. Перенос делался через robocopy
(rename напрямую не сработал — папку держали процессы MCP-сервера), скопировано 8400 файлов,
0 расхождений. `.venv` пересоздан заново (`uv sync`) вместо копирования — в нём зашиты
абсолютные пути. MCP-сервер перерегистрирован на новый путь, `claude mcp list` → Connected.
Старая папка `knowledge-factory/` частично осталась на диске (48 заблокированных файлов
внутри бывшего `.venv`) — освободится после перезагрузки Windows, можно будет удалить руками.

**Важно для Docker:** у пользователя старая версия Windows, Docker Desktop работает через
Hyper-V (не WSL2) — менее стабилен с проброшенными портами, особенно после перезапуска хоста.
Если контейнер не отвечает — не пытаться чинить бесконечно самому, сообщить пользователю
(правило от него), либо попробовать `docker restart <контейнер>` — часто помогает.
