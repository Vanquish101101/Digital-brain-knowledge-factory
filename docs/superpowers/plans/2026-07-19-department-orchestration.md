# Оркестрация работы по отделам — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Задокументировать протокол оркестрации работы по отделам (Вариант A из спеки) —
создать общий журнал операций и добавить в `CLAUDE.md` короткую ссылающуюся секцию, без
изменений кода/тестов.

**Architecture:** Чисто документационная задача — два новых/изменённых markdown-файла. Нет
кода, нет тестов в традиционном смысле; «проверка» каждого шага — визуальная сверка
содержимого файла со спекой.

**Tech Stack:** Markdown-файлы в существующей структуре проекта (`project-config/`,
корневой `CLAUDE.md`).

## Global Constraints

- Полная спека: `docs/superpowers/specs/2026-07-19-department-orchestration-design.md` —
  каждый task ниже реализует конкретный раздел спеки, значения (названия файлов, состав
  отделов, пороги) копируются из неё дословно.
- Формат коммитов: `Digital brain | <описание> | V X.X.X`, следующая версия — **V1.4.67**
  (после этого плана — V1.4.68 для второй задачи, если коммитятся раздельно; в конце
  зафиксировать финальный номер).
- Работа напрямую в `main`, без отдельной ветки/worktree (установленная конвенция проекта).
- Перед `git push` — всегда спросить пользователя (установленная конвенция проекта).
- Личные файлы пользователя (`Журнал знаний.md`, `Разработкаи Обращение к АИ/*`) никогда не
  трогать и не добавлять в `git add`.

- Produces: файл с фиксированной структурой (заголовок + правило владения + формат записи +

- [ ] **Step 1: Создать файл с полным содержимым**

Записать в `E:\Digital brain\project-config\АГЕНТЫ-ЛОГ.md` ровно следующий текст:

```markdown
# Агенты-лог: сквозной журнал операций по отделам

Единый сквозной журнал работы агентов (суб-агентов и моей собственной) по всем отделам
проекта — не привязан к одной фиче или плану. Полное описание протокола — в
`docs/superpowers/specs/2026-07-19-department-orchestration-design.md`.

**Отличие от других логов проекта:**
- `Журнал знаний.md` (в корне) — про контент `raw-data-repository/` (что добавлено/изменено/
  удалено в vault), не про работу агентов. Не путать.
- `.superpowers/sdd/progress.md` — детальный реестр по одной активной фиче/плану,
  git-ignored, временный. Остаётся как есть, не заменяется этим файлом.
- Этот файл — высокоуровневый, сквозной, переживает отдельные фичи и сессии.

**Правило владения:** в этот файл пишу только я (главный оркестратор), после получения
отчёта от суб-агента или после самостоятельно выполненной работы. Суб-агенты никогда не
пишут в него напрямую — это защита от конфликта одновременной записи.

**Формат одной записи:**

```markdown
## ГГГГ-ММ-ДД — [Отдел] — [Краткое название задачи]
- Кто выполнял: я напрямую / суб-агент (implementer/reviewer/...)
- Результат: успех / сбой / эскалация
- Отчёт: ссылка на report-файл, если был
- Примечание: 1-2 предложения по сути, если нужно
```

---

## 2026-07-19 — Мета — Создание протокола оркестрации по отделам

- Результат: успех

- Примечание: первая запись в этом журнале — сам факт формализации протокола (спека →
  план → этот файл). Дальнейшие записи — по мере реальной работы с отделами.
```

- [ ] **Step 2: Вставить новую секцию**

Вставить перед `## Отчёт по итогам задачи` ровно следующий текст (с пустой строкой до и
после):

```markdown
## Оркестрация работы по отделам

Работа со скиллами, установленными по тематическим «отделам» (веб, продукт, аналитика,
маркетинг — см. `карта_бз.md`/git-историю установки скиллов), следует протоколу из
`docs/superpowers/specs/2026-07-19-department-orchestration-design.md`. Кратко:

- **Отделы:** Веб (`frontend-design`, `web-design-guidelines`, `vercel-react-best-practices`,
  `shadcn`, `site-architecture`, `schema`, `cro`), Продукт (`nodejs-backend-patterns`,
  `fastapi-templates`, `docker-compose-orchestration`, `postgresql-table-design`,
  `qdrant-clients-sdk`, `qdrant-search-quality`, `redis-core`, `onboarding`, `signup`,
  `paywalls`), Аналитика (`analytics`, `ab-testing`), Маркетинг (`copywriting`,
  `copy-editing`, `seo-audit`, `product-marketing`, `customer-research`,
  `marketing-psychology`, `content-strategy`, `ai-seo`, `pricing`, `social`, `emails`,
  `referrals`, `launch`).
- **Роли:** я — главный оркестратор всегда; оркестратор отдела — я же напрямую для рутинных
  задач, отдельный суб-агент-координатор — только для по-настоящему объёмных многоагентных
  задач внутри отдела.
- **Общая память:** каждая операция логируется в `project-config/АГЕНТЫ-ЛОГ.md` (пишу только
  я, суб-агенты не пишут напрямую).
- **Отказоустойчивость:** при сбое суб-агента — один автоматический ретрай со свежим
  контекстом, дальше — беру задачу на себя или эскалирую пользователю, не бесконечный retry.
- **Консенсус:** пользователь > план (если ревью не находит в нём конкретного дефекта) >
  вердикт ревьюера > самооценка исполнителя; конфликт ревью vs план — всегда к пользователю.
- **Проверка:** максимум 2 цикла «найдено → исправлено → ревью» на задачу, дальше —
  эскалация, не бесконечный цикл.
- **Эскалация — в реальном времени**, в момент возникновения (не собирается в финальный
  отчёт), но только при реальном пересечении порога, не на каждый шаг.

Разработка самого `knowledge-factory` (`kf.py`) — отдельный, уже существующий цикл
(`brainstorming` → `writing-plans` → `subagent-driven-development`), этим протоколом не
затрагивается.
```
