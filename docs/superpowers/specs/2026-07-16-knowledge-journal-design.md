# Дизайн: журнал знаний + регламент маршрутизации

Дата: 2026-07-16
Статус: одобрено пользователем, готово к плану реализации

## Контекст

В проекте уже есть:
- `raw-data-repository/` — человекочитаемый vault (Obsidian-подобный), с готовой структурой
  разделов (001 Входящие … 010 Архив, включая `007 Проекты/` с подпапками по темам —
  Веб Дизайн, Видео Дизайн, Аудио дизайн, 3D Моделирование, Game Дизайн, Маркетинг).
- `knowledge-factory/` — Docker-бэкенд + `kf.py` (ingest/search/ask/stats): хэш-дедупликация
  через Postgres (`needs_ingest`/`record_ingested`), автоматический слой синтез-заметок
  поверх каждого проиндексированного файла (`kf/synthesize.py` + `kf/ingest.py`).
- `карта_бз.md` — структурная карта проекта верхнего уровня, с таблицей "Куда класть новое"
  (по типу материала), обновляется вручную при изменениях папок/файлов верхнего уровня.
- `CLAUDE.md` — правило "Регламент автодобавления знаний": куда положить материал, когда
  пользователь просит "добавь в базу знаний".

Пользователь хочет:
1. Присылать материалы в любом формате (PDF, docx, excel, ссылки, ссылки на видео, статьи,
   текст, видео, фото, аудио, голосовые сообщения, изображения и т.д.) и получать
   подтверждение, куда именно они легли.
2. Чёткий регламент маршрутизации не только по типу материала, но и **по теме**: материал
   про Маркетинг → в раздел/проект, связанный с Маркетингом; про 3D → в 3D; про Веб-дизайн →
   в Веб-дизайн — по аналогии для всех существующих направлений/хобби/проектов.
3. Durable "журнал знаний" — при добавлении или удалении любого материала в
   `raw-data-repository/` (в том числе вручную, напрямую в Obsidian, без участия ассистента)
   в журнал автоматически попадает короткая запись: что произошло, где, когда, о чём.

## Требования

### Регламент маршрутизации

Расширяется существующая таблица "Куда класть новое" в `карта_бз.md` (единственный источник
правды, на который ссылается `CLAUDE.md`):

- Правила **по типу материала** остаются как есть (сырое/не разобрано → Входящие, конспект →
  Знания, курс/книга → Ресурсы и т.д.) — это первый фильтр, когда тема неочевидна.
- Добавляются правила **по теме**: если материал явно относится к одной из существующих
  тем `007 Проекты/` (Веб-дизайн, Видео-дизайн, Аудио-дизайн, 3D, Game-дизайн, Маркетинг) или
  к `008 Интересы (Увлечения)` — кладём туда, независимо от формата файла (PDF о 3D → в
  3D-раздел, а не в общий Ресурсы).
- Если тема не совпадает ни с одним существующим разделом/проектом/хобби — действует старое
  правило по умолчанию: `003 Знания` (конспекты/наблюдения), `005 Ресурсы` (курсы/книги/
  референсы), а при полной неясности — `001 Входящие`.
- Формат (PDF/docx/excel/видео/аудио/голосовое/фото и т.д.) определяет **как** материал
  обрабатывается технически (текст читается напрямую, видео — транскрипция + разбор кадров,
  изображение — OCR/vision-caption и т.д. — это уже реализовано в `kf/extract.py`), а тема —
  **куда** он кладётся. Это два независимых измерения одного решения.

### Журнал знаний

Новый файл `Журнал знаний.md` в **корне проекта** (рядом с `карта_бз.md`), не внутри
`raw-data-repository/` — чтобы не попадать в собственную индексацию и не создавать
бесконечный цикл переиндексации.

Каждая запись в журнале:
```
- 2026-07-16 | добавлено | raw-data-repository/005 Ресурсы/курс-3d.pdf | Ресурсы | О чём материал (первая фраза синтез-заметки)
- 2026-07-16 | удалено | raw-data-repository/003 Знания/старая-заметка.md | Знания | (без описания)
```

Механизм пополнения — привязан к `kf.py ingest`, не к ручным действиям ассистента:
- **Добавления/изменения** — переиспользуется существующий хэш-механизм `needs_ingest`.
  Для каждого реально проиндексированного файла в журнал попадает путь, раздел (первый
  сегмент пути внутри `raw-data-repository/`) и краткое описание, извлечённое из уже
  сгенерированной синтез-заметки (без дополнительного вызова LLM) по правилу: взять
  первую непустую строку текста заметки, убрать ведущую нумерацию/маркдаун-разметку
  (`"1. "`, `"- "` и т.п.), обрезать до 150 символов. Если текст заметки пуст (синтез не
  удался) — описание в журнале не пишется, только путь/раздел/дата.
- **Удаления** — новая логика: сравниваются все пути оригинальных (не синтез-заметочных)
  документов, известные Postgres, с реально найденными файлами при текущем скане
  `raw-data-repository/`. Отсутствующие — считаются удалёнными и логируются.
  **Векторы/строки в Qdrant/Postgres/MinIO при этом не удаляются** — это осознанно
  отложенное решение (см. ниже), ассистент задаёт вопрос пользователю в диалоге при
  обнаружении такой записи.
- Журнал пополняется независимо от того, кто добавил/удалил файл — сам пользователь через
  ассистента или напрямую в Obsidian — поскольку триггер один: любой запуск `kf.py ingest`.

Практическое следствие для рабочего процесса ассистента: после того как я кладу новый файл
в `raw-data-repository/`, я должен сразу запустить `kf.py ingest`, чтобы файл и попал в
векторный поиск, и появился в журнале — это одно действие, не два отдельных шага.

## Архитектура

- Новый модуль `kf/journal.py`:
  - `journal_path(settings) -> Path` — путь к `Журнал знаний.md` в корне проекта.
  - `format_entry(action, path, section, description, date) -> str` — форматирует одну
    строку записи.
  - `detect_deleted(known_paths: set[str], seen_paths: set[str]) -> set[str]` — чистая
    функция, разница множеств (без обращения к БД — принимает уже готовые множества).
  - `append_entries(entries: list[str], path: Path) -> None` — дописывает готовые строки
    в конец файла (создаёт файл с заголовком, если не существует).
- Изменения в `kf/ingest.py::ingest_directory()`:
  - При обработке каждого файла — собирать `(action, rel_key, section, description)` для
    журнала вместо немедленной записи (накопление в списке).
  - После основного цикла — запросить у Postgres все пути оригинальных документов
    (исключая пути с префиксом `synthesis_notes_dir.name`), сравнить с множеством путей,
    реально увиденных в этом скане, вызвать `detect_deleted`, добавить записи "удалено".
  - В конце — одним вызовом `append_entries` дописать все собранные записи в журнал (если
    список не пуст).
- `kf/cli.py`: вывод `ingest` дополняется количеством новых записей в журнале и, если были
  обнаружены удаления — явным предупреждением в консоли (`⚠ Обнаружены удалённые файлы: …`),
  чтобы это было заметно сразу, а не только в файле журнала.

## Обработка ошибок

- Ошибка записи в `Журнал знаний.md` (например, файл занят/нет прав) не должна прерывать
  весь `ingest` — только предупреждение в консоль, по аналогии с уже существующей
  обработкой ошибок синтез-заметок (`notes_failed`).
- Ошибка при запросе списка путей для detect_deleted (например, БД недоступна) — тоже
  не должна прерывать ingest, деградируем до "удаления в этом запуске не проверялись".

## Тестирование

- `kf/journal.py` — чистые юнит-тесты (`detect_deleted`, `format_entry`, `append_entries`
  через `tmp_path`), без сети/БД, по образцу существующих тестов проекта.
- `kf/ingest.py` — расширение существующих тестов `test_ingest.py` моками Postgres/Qdrant,
  как уже сделано для синтез-заметок в предыдущей итерации.

## Вне рамок (явно)

- **Автоматическая очистка Qdrant/Postgres/MinIO для удалённых файлов** — открытый вопрос,
  осознанно отложен пользователем 2026-07-16 (сохранено в памяти ассистента). Реализуется
  отдельно, после решения.
- **Мгновенное отслеживание через файловый watcher** — журнал обновляется только при
  запуске `kf.py ingest`, не в реальном времени.
- **Обратное заполнение журнала историей уже существующих 23 документов** — журнал стартует
  "с нуля" со следующего запуска `ingest`; существующий контент задним числом не логируется
  (можно сделать отдельно по запросу).
- **Разделы для тем, которых ещё не существует в `raw-data-repository/`** — обрабатывается
  по уже существующему правилу неопределённости (Знания/Ресурсы/Входящие по умолчанию),
  создание новых тематических подпапок — отдельное решение пользователя, не автоматическое.
