# Синхронизация удалений (kf.py sync-deletions) — дизайн

Дата: 2026-07-19

## Проблема

Сейчас, когда файл удаляется из `raw-data-repository/`, `kf.py ingest` это обнаруживает и
пишет запись «удалено» в `Журнал знаний.md`, но векторы/записи в Qdrant/Postgres/MinIO не
удаляются — они остаются в базе знаний навсегда, ссылаясь на файл, которого больше нет.

## Принцип, из которого исходим

`raw-data-repository/` — единственный источник истины о том, что пользователь сам решил
сохранить. Векторное хранилище — производная от него, а не независимая память. Если
пользователь сам удалил файл из vault — это уже его осознанное решение, и держать осиротевшие
векторы вечно не помогает (это не «память», а технический мусор). Но это не должно быть
слепой автоматикой: переименование файла в Obsidian выглядит идентично удалению (старый путь
пропал), и должно быть отличено, а не спровоцировать ненужную повторную обработку.

## Цели

- Новая команда `kf.py sync-deletions [--yes]`. Без `--yes` — только показывает план (что
  будет очищено, что похоже на переименование и поэтому пропущено). С `--yes` — реально
  выполняет очистку.
- Различение «реально удалено» и «переименовано/дублировано» — по сравнению sha256-хэша
  пропавшего пути с хэшами всех путей, которые всё ещё существуют. Совпадение = контент жив
  под другим именем, не трогаем.
- Очистка охватывает: исходный файл и его синтезированную LLM-заметку, во всех хранилищах —
  Postgres, **все профили эмбеддинга в Qdrant** (не только активный — иначе при возврате на
  ранее использованную модель удалённый контент всплывёт снова), MinIO (включая резервную
  копию исходного файла, которая переживает удаление из vault), граф знаний (Kuzu), и
  физический файл синтезированной заметки на диске.
- Запись в `Журнал знаний.md` о факте очистки (отдельное действие от уже существующего
  «удалено», которое пишет `kf.py ingest` при обнаружении пропажи).

## Не цели (явно не делаем в этой итерации)

- **Не делаем контентную «уборку» знаний** — обзор базы на предмет устаревшего/избыточного/
  контрпродуктивного контента, который физически никуда не делся, но мог потерять
  актуальность. Это отдельный, более крупный будущий модуль (обсуждался тем же вечером,
  зафиксирован в памяти ассистента как `project-content-curation-module-future`) — там
  ничего не удаляется автоматически по содержанию, только по явному решению пользователя
  после ревью. Эта фича (`sync-deletions`) — только про синхронизацию с уже принятым
  пользователем решением удалить файл из vault, не про суждение о содержании.
- Не встраиваем автоматический запуск `sync-deletions` внутрь обычного `kf.py ingest` —
  остаётся отдельной, осознанно вызываемой командой.
- Не пытаемся распознать переименование внутри одной *папки* специальным образом — сравнение
  по хэшу работает одинаково для перемещения между разделами и переименования на месте.

## Архитектура

### 1. `kf/store/postgres.py` — новая функция

`list_paths_with_hashes(conn, exclude_prefix: str = "") -> dict[str, str]` — как уже
существующий `list_paths`, но возвращает словарь путь→sha256 вместо множества путей.
Аддитивно, по образцу уже существующего `list_paths`.

### 2. Новый модуль `kf/deletion_sync.py`

Чистая логика без побочных эффектов (сравнима с уже существующим `kf/journal.py`):

- `compute_deletion_candidates(known_source_hashes: dict[str, str], seen_source_paths: set[str], all_known_hashes: dict[str, str]) -> tuple[list[str], list[str]]`
  — возвращает `(confirmed_deleted, likely_renamed)`, оба отсортированы. Путь из
  `known_source_hashes`, которого нет в `seen_source_paths` — кандидат на удаление; если его
  хэш совпадает с хэшем любого пути, который **есть** в `seen_source_paths` (ищем по
  `all_known_hashes`) — это переименование, не удаление.
- `note_rel_key_for(source_rel_key: str, notes_dir_name: str) -> str` — тот же паттерн
  построения ключа заметки, что уже используется в `kf/ingest.py`
  (`f"{notes_dir_name}/{source_rel_key}.md"`), вынесенный в переиспользуемую функцию.

Побочные эффекты — отдельная функция в том же модуле:

- `purge_source(source_rel_key: str, settings: Settings, pg_conn, qdrant_client, minio_client, graph_conn) -> None`
  — удаляет исходный файл и его заметку из Postgres (`DELETE FROM documents WHERE path IN
  (...)`), из **каждой** коллекции в `EMBEDDING_PROFILES` (`kf.store.qdrant_store.delete_by_path`),
  из MinIO (`remove_object` на бакете `raw`, оба ключа), из графа
  (`kf.store.graph_store.delete_relationships_by_source` для исходного пути — извлечение
  сущностей всегда ведётся от исходного текста, не от заметки), и удаляет физический файл
  заметки на диске, если он есть. Каждый шаг — в своём `try/except` с логом, чтобы сбой
  одного хранилища не остановил очистку остальных (тот же принцип отказоустойчивости, что
  уже применён в `kf/ingest.py`).

### 3. Команда `kf.py sync-deletions [--yes]`

Сканирует `raw-data-repository/` (тот же обход, что и `ingest_directory`, через
`should_index`), сравнивает с Postgres, вызывает `compute_deletion_candidates`. Без `--yes`
печатает план и останавливается. С `--yes` — вызывает `purge_source` для каждого
подтверждённого удаления, затем дописывает в `Журнал знаний.md` запись с действием
`очищено_из_базы` для каждого через уже существующие `format_entry`/`append_entries`.

## Риски и допущения

- Хэш-сравнение ловит точные дубликаты содержимого как «переименование», даже если пути
  реально не связаны (два разных файла с одинаковым текстом) — на практике для личной базы
  знаний это редкий и не опасный случай (в худшем случае лишний раз не почистится то, что
  реально стоило почистить — не потеря данных, а консервативная осторожность).
- Как и остальные операции с реальными хранилищами в этом проекте, `sync-deletions --yes`
  не покрывается юнит-тестами построчно на каждое хранилище — логика `compute_deletion_candidates`
  тестируется как чистая функция; `purge_source`/сама команда проверяются вручную на реальных
  данных при финальной верификации, тем же способом, что и предыдущие фичи этой сессии.
