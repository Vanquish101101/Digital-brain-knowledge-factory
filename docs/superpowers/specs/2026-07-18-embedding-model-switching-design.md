# Переключаемые модели эмбеддинга — дизайн

Дата: 2026-07-18

## Проблема

Сейчас модель эмбеддинга захардкожена (`sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`,
локально через fastembed, размерность вектора 384). Пользователь хочет иметь возможность
быстро переключаться между локальной моделью и облачными моделями через OpenRouter
(Qwen3-Embedding-8B, OpenAI text-embedding-3-small/large), не переписывая код и не гадая,
что делать со старыми векторами при смене модели.

Ключевая техническая сложность: у разных моделей разная размерность вектора, а коллекция в
Qdrant жёстко привязана к одной размерности при создании. Значит "переключение модели" — это
не просто смена настройки, а вопрос хранения и актуальности векторов разных моделей
одновременно.

## Цели

- Несколько заранее описанных профилей эмбеддинга (локальный + через OpenRouter), переключение
  между которыми — одна CLI-команда.
- Переключение на уже использовавшуюся ранее модель — мгновенное, без затрат API.
- Первое использование новой модели или расхождение данных после переключения — требует
  явного, отдельного, дешёвого по возможности шага (не происходит само по себе и не тратит
  деньги без ведома пользователя).
- Существующие данные (текущая коллекция Qdrant на MiniLM) не теряются и не мигрируют без
  необходимости — фича должна быть накатываемой без риска для уже работающей базы.

## Не цели (явно не делаем в этой итерации)

- Не делаем автоматический реиндекс при переключении модели.
- Не делаем UI/кнопку — CLI-команда и есть "кнопка".
- Не трогаем граф знаний (Kuzu) и LLM-синтез заметок — они не зависят от модели эмбеддинга.
- Не добавляем произвольный конфиг моделей (YAML и т.п.) — список моделей фиксирован в коде,
  расширяется правкой одного файла.

## Архитектура

### 1. Реестр моделей — `kf/embedding_models.py`

Фиксированный словарь профилей в коде (не конфиг-файл — проект нигде не использует внешние
конфиги, всё через `.env` + код):

- `local` → provider=`local`, model_id=`sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`,
  dimension=384, collection=`knowledge` (**совпадает с текущим именем коллекции** — профиль
  `local` не мигрирует данные, а просто продолжает использовать то, что уже есть)
- `qwen3-8b` → provider=`openrouter`, model_id=`qwen/qwen3-embedding-8b`, dimension=4096
  (уточнить эмпирически при реализации — по документации Qwen3-Embedding-8B поддерживает
  диапазон 32–4096, берём максимум по умолчанию), collection=`knowledge__qwen3_8b`
- `openai-small` → provider=`openrouter`, model_id=`openai/text-embedding-3-small`,
  dimension=1536, collection=`knowledge__openai_small`
- `openai-large` → provider=`openrouter`, model_id=`openai/text-embedding-3-large`,
  dimension=3072, collection=`knowledge__openai_large`

Добавление новой модели в будущем = одна новая запись в этом словаре, без изменений в
остальном коде.

### 2. Состояние "активная модель сейчас"

Текстовый файл `knowledge-factory/data/active_embedding_model.txt`, содержит только имя
профиля (например `qwen3-8b`). Отсутствие файла = профиль `local` (поведение по умолчанию,
полностью обратно совместимо с текущим состоянием проекта). Команда `embedding-model use`
перезаписывает файл; никакой ручной правки `.env` не требуется.

### 3. Провайдер эмбеддинга через OpenRouter

В `kf/embeddings.py` добавляется вызов `POST https://openrouter.ai/api/v1/embeddings` через
`httpx`, тем же паттерном, что уже используется в `kf/graph.py` для LLM-вызовов (тот же
`Bearer`-токен `settings.openrouter_api_key`). Модуль предоставляет единую точку входа для
получения векторов, которая внутри диспетчеризуется по `provider` активного профиля
(`local` → fastembed, `openrouter` → HTTP-вызов) — вызывающий код (`ingest.py`, `api.py`) не
знает и не должен знать, какая модель активна.

### 4. Новые CLI-команды: `kf.py embedding-model ...`

- **`list`** — показать все профили из реестра, отметить активный, для каждого вывести
  "покрытие": сколько путей из Postgres (`list_paths`) реально присутствует в его коллекции
  Qdrant. Проверка бесплатна — это сверка уже существующих метаданных, без обращения к
  внешним API.
- **`use <имя>`** — переключить активный профиль (переписать файл состояния) и сразу вывести
  предупреждение, если коллекция выбранного профиля отстаёт от текущего состава базы (тот же
  бесплатный расчёт покрытия, что и в `list`).
- **`sync`** — досчитать эмбеддинги только для путей, отсутствующих в коллекции активного
  профиля. Использует уже извлечённый текст файла (без повторного OCR/транскрибации), только
  считает вектор и записывает в Qdrant. **Не запускает** LLM-синтез заметок и извлечение
  сущностей графа — это единственная операция группы, которая тратит API-бюджет (если активный
  профиль не `local`), и делает это по минимуму.

### 5. Интеграция с существующими командами

`open_session()` (`kf/api.py`) и `_build_ingest_deps()` (`kf/cli.py`) перестают использовать
захардкоженные `COLLECTION = "knowledge"` / `VECTOR_SIZE = 384` — читают активный профиль из
файла состояния и резолвят имя коллекции и размерность из реестра. `kf.py search`, `kf.py ask`
и обычный `kf.py ingest` автоматически работают через модель, которая выбрана на момент
запуска — без дополнительных флагов и изменений в их собственном коде.

Для расчёта "покрытия" в Qdrant нужна новая функция перечисления уникальных путей в коллекции
(`kf/store/qdrant_store.py`, через `scroll`), аналогичная уже существующей
`list_paths` в `kf/store/postgres.py`.

### 6. Обработка ошибок

Сбой сетевого вызова к OpenRouter при эмбеддинге (в `sync` или в обычном `ingest`, если
активна облачная модель) — лог и пропуск конкретного файла, без остановки всей операции.
Паттерн идентичен уже существующей обработке сбоя извлечения сущностей графа в `ingest.py`
(`_extract_and_store_entities`).

### 7. Тестирование

Юнит-тесты без сети: сборка запроса к OpenRouter embeddings API и разбор ответа, реестр
профилей (наличие всех полей, уникальность имён коллекций), расчёт "покрытия" (разница
множеств путей). Сам HTTP-вызов остаётся непокрытым юнит-тестами — как и текущий вызов
LLM в `kf/graph.py` — и проверяется вручную на реальных данных при финальной верификации
фичи, тем же способом, что и граф знаний.

## Риски и допущения

- Точная размерность вектора Qwen3-Embedding-8B по умолчанию (4096) — предположение по
  документации, требует подтверждения первым реальным вызовом при реализации; если
  фактическая размерность отличается, значение в реестре надо поправить до создания коллекции.
- Переключение на профиль с "молодой" (пустой или частично заполненной) коллекцией не ломает
  `search`/`ask` — они просто вернут меньше или ноль результатов, что ожидаемо и будет видно
  пользователю через предупреждение о покрытии в `list`/`use`.
