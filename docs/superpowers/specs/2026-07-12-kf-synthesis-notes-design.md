# Расширение kf.py: слой синтез-заметок поверх обычного ingest

Дата: 2026-07-12
Статус: одобрено пользователем, готово к написанию implementation-плана

## Контекст и цель

Пользователь хочет подключать внешние диски (в основном — обучающие курсы и
материалы, много видео, где важен не только звук, но и визуальный ряд),
запускать анализ через `kf.py`, затем отключать диск — и чтобы полученное
понимание материала оставалось в Digital Brain навсегда, без необходимости
повторного подключения источника для повторного анализа.

Сегодня `kf.py ingest` уже извлекает *дословный* текст (транскрипт речи,
OCR, подписи к кадрам — см. `docs/superpowers/specs/2026-07-12-kf-image-video-ingest-design.md`)
и режет его на чанки для семантического поиска. Это точный, но
неструктурированный слой: сырой транскрипт часовой лекции — это скомканная
стенограмма, а не понимание. Пользователь хочет добавить второй,
осмысленный слой — по аналогии с тем, как в памяти человека при возникновении
задачи «подтягивается» связанный по смыслу контекст, а не сырые дословные
воспоминания.

## Требования (из брейнсторминга с пользователем)

- Ничего в существующем, уже проверенном пайплайне `kf.py ingest` не
  меняется и не ломается: чанки сырого текста, копия файла в MinIO — всё
  остаётся как есть.
- Поверх этого добавляется новый, отдельный, необязательный по своей природе
  слой: LLM читает извлечённый текст файла целиком и пишет структурированную
  заметку — о чём материал, ключевые идеи, как это может пригодиться в
  будущих задачах.
- Синтез делается для **всех** типов файлов без исключений (текст, фото,
  видео, PDF) — не только для видео/картинок, где сырой текст особенно
  неструктурирован.
- Заметки складываются в отдельную новую папку внутри
  `Цифровой мозг (digital-brain)/`, рядом с `data/` — не смешиваясь ни с
  vault (`Хранилище входных данных/`), ни с существующими данными.
  Пользователь ещё не уверен в итоговой форме этого слоя, поэтому важно,
  чтобы его было легко пересмотреть/пересобрать заново, не трогая основной
  vault.
- Заметки тоже должны быть доступны через `semantic_search`/`ask` — иначе
  эффект «подтягивания связанного контекста» не работает, заметки просто
  лежат файлами.
- Работает полностью автоматически, встроено в `kf.py ingest` (один запуск —
  и чанки, и заметки готовы), а не отдельной командой. Согласуется с уже
  принятым ранее в проекте принципом «полностью автоматически» для
  картинок/видео.

## Архитектура: новый модуль + расширение ingest

| Модуль | Функция | Назначение |
|---|---|---|
| `kf/synthesize.py` (новый) | `synthesize_note(settings, text, source_path) -> str` | HTTP-вызов к LLM через OpenRouter (по образцу `kf/llm.py::call_llm`, тот же `LLM_MODEL`, что и для `kf.py ask`) — читает извлечённый текст, возвращает структурированную заметку |

`kf/ingest.py` дополняется: после того как обычные чанки файла успешно
записаны (без изменений в этой части), для файлов вне
`Синтезированные данные (synthesized-notes)/` вызывается `synthesize_note()`,
результат пишется как `.md`-файл в эту папку под тем же `rel_key`, что уже
используется в `ingest_directory` для MinIO-ключа и Qdrant/Postgres-пути
(`path.relative_to(source_dir)` — путь относительно `--source`, с добавленным
`.md`), и эта заметка сразу же доиндексируется тем же самым механизмом
чанкинга — как обычный текстовый файл (`extract_text` читает `.md` напрямую,
без OCR/whisper). Переиспользование `rel_key` — сознательный выбор: у
одноимённых файлов из разных источников (`--source`) уже сегодня совпадает
идентичность в MinIO/Qdrant, это не новое ограничение, вносимое синтез-слоем.

`kf/config.py` получает новую настройку `SYNTHESIS_NOTES_DIR` (по умолчанию
`Цифровой мозг (digital-brain)/Синтезированные данные (synthesized-notes)/`).
Отдельная модель для синтеза не заводится — переиспользуется уже
существующий `LLM_MODEL`.

## Поток данных (пример)

```
D:\курс\lesson3.mp4  (запуск: kf.py ingest --source D:\курс)

  [без изменений]
  extract_text() → транскрипт (Whisper) + OCR/vision по кадрам
  → чанки → embeddings → Qdrant + Postgres
  → копия файла → MinIO

  [новое, добавочно]
  извлечённый текст выше → synthesize_note() → LLM-заметка
  → записывается как (rel_key = "lesson3.mp4", тот же, что и в MinIO/Qdrant):
    "Цифровой мозг (digital-brain)/Синтезированные данные (synthesized-notes)/lesson3.mp4.md"
  → эта .md-заметка сама проходит обычный ingest (без OCR/whisper)
  → тоже становится чанками в Qdrant + Postgres
```

После этого `kf.py ask "чему был посвящён курс?"` находит и сырые чанки
транскрипта, и (обычно с более высоким скором релевантности за счёт более
ёмкого текста) — синтез-заметку.

## Обработка ошибок

Синтез — необязательный дополнительный шаг: его сбой не должен портить уже
успешно сделанную обычную индексацию файла. Вызов `synthesize_note()`
оборачивается в отдельный `try/except` (по аналогии с уже принятой в проекте
схемой обработки ошибок на уровне файла — `files_failed` в `kf/ingest.py`).
При сбое (сеть, лимит OpenRouter и т.п.) — печатается предупреждение с именем
файла, инкрементируется новое поле `notes_failed` в `IngestStats`, обработка
переходит к следующему файлу. Сырые чанки файла при этом остаются в базе —
они уже были успешно записаны до попытки синтеза.

Итоговая сводка `kf.py ingest`:
```
Готово. просканировано: N, проиндексировано: N, пропущено: N, ошибок: N,
чанков записано: N, заметок синтезировано: N, ошибок синтеза: N
```

Защита от зацикливания: файлы, уже лежащие внутри
`Синтезированные данные (synthesized-notes)/`, не отправляются на повторный
синтез (не пишем заметку про заметку).

## Тестирование

По уже принятому в проекте прецеденту (`caption_image` в
`kf/vision_caption.py`, `call_llm`/`ask`): сам вызов LLM в
`synthesize_note()` — реальный платный запрос к OpenRouter, автотестом не
покрывается, только ручной проверкой на реальных файлах.

Автотестами (TDD, с моками LLM-вызова) покрывается логика оркестрации в
`kf/ingest.py`:
- после успешной индексации файла вызывается `synthesize_note()`, результат
  пишется как `.md` в `Синтезированные данные (synthesized-notes)/`, путь
  повторяет относительный путь исходника;
- эта `.md`-заметка сама доиндексируется (появляется в чанках);
- файлы внутри `Синтезированные данные (synthesized-notes)/` синтез
  пропускают;
- сбой `synthesize_note()` не прерывает обработку файла, инкрементирует
  `notes_failed`, обычные чанки файла остаются в базе.

Плюс ручная сквозная проверка на реальных файлах (текст/фото/видео/PDF) с
замером реальной стоимости и времени на паре файлов, прежде чем гонять на
всём внешнем диске целиком.

## Вне рамок этой задачи

- **Копия сырого файла в MinIO** — поведение не меняется, остаётся как есть
  для любого источника (в том числе внешних дисков). Решение принято явно:
  не трогать существующую проверенную архитектуру.
- **Сэмплирование кадров в длинных видео** (часовые лекции — сейчас
  визуальный ряд покрывает только первые ~5 минут при лимите
  `MAX_VIDEO_FRAMES`) — вопрос сознательно отложен. Пользователь планирует
  позже интегрировать сюда модуль глубокого парсинга из отдельного
  «Marketing agency Project» (4 агента, изначально разрабатывался в том
  числе под задачи качественного понимания любого типа контента, включая
  видео без звука) как отдельный связанный модуль — не переизобретать это
  средствами `kf.py` сейчас.
- **Отдельная LLM-модель для синтеза** — переиспользуется существующий
  `LLM_MODEL`; отдельная настройка не заводится, можно добавить позже при
  необходимости.
