# Оркестрация работы по отделам — дизайн (Вариант A)

## Контекст и цель

По просьбе пользователя формализуем, как именно я (Claude) организую работу внутри проекта
Digital Brain, когда она затрагивает один или несколько «отделов» — тематических групп
скиллов, установленных в этой же сессии (веб, продукт, аналитика, маркетинг). Основа —
6 принципов масштабирования мультиагентных систем из лекции (Ержан, неделя 6, транскрипт в
`project-config/Транскрибированная лекция видео/АРХИТЕКТУРА_АГЕНТА_6_ПРИНЦИПОВ_МАСШТАБИРОВАНИЯ_MAS_Транскрипация.docx`).

Цель — не построить новую инфраструктуру, а достроить то, чего не хватает поверх уже
работающего цикла `superpowers:subagent-driven-development` (task-brief → report,
task-reviewer, финальное ревью). Два принципа из шести уже реализованы этим циклом
(«Передача дел», «Проверка») — их не переделываем, только документируем как есть и добавляем
один явный порог. Четыре принципа («Общая память», «Отказоустойчивость», «Консенсус», «Чат
агентов») формализуются заново.

Явно отклонённая альтернатива: реальная интеграция `ruflo` (`ruvnet/ruflo`, 314+ MCP-
инструментов для памяти/хуков/роутинга/суб-агентов) — технически более полное решение,
буквально совпадающее с лекцией, но требующее отдельной настройки MCP-сервера (инструменты
сейчас не подключены к сессии, хотя в глобальном `CLAUDE.md` есть блок, будто интеграция уже
настроена — это расхождение, отдельная тема). Пользователь явно попросил напомнить об этом
варианте («Вариант Б») в начале следующей новой сессии — зафиксировано в памяти ассистента
(`project_ruflo_full_integration_reminder.md`), не является частью этой спеки.

## Роли

- **Главный оркестратор** — я, всегда, на верхнем уровне любой сессии.
- **Оркестратор отдела** — роль, не обязательно отдельный суб-агент. Для задачи в пределах
  одного отдела эту роль исполняю я сама напрямую: загружаю нужные скиллы отдела, слежу за
  соблюдением принципов 2–6 ниже. Отдельный суб-агент-координатор отдела дисптачится только
  для по-настоящему объёмных многоагентных задач внутри отдела (по аналогии с текущим циклом
  разработки `kf.py`) — не для рутинных задач вроде установки скилла или короткой правки.
- **Исполнители** — обычные дисптачнутые суб-агенты (implementer, task-reviewer и т.д.),
  роли не меняются относительно уже используемого `subagent-driven-development`.

## Принципы

### 1. Передача дел — без изменений

Контракт `task-brief` (полные требования задачи) → `report`-файл (что сделано, коммиты,
результаты тестов, замечания) между агентами, как уже реализовано в
`subagent-driven-development`. Тест на полноту передачи (из лекции): если принявший агент
начинает переспрашивать — значит, передача была неполной.

### 2. Общая память (новое)

Новый файл: `project-config/АГЕНТЫ-ЛОГ.md` — единый сквозной журнал операций **по всем
отделам**, не привязан к одной фиче/плану.

Отличие от уже существующих файлов:
- `Журнал знаний.md` — про контент `raw-data-repository/` (что добавлено/изменено/удалено в
  vault), не про работу агентов. Не путать.
- `.superpowers/sdd/progress.md` — детальный реестр по одной конкретной фиче/плану, локальный
  для активной разработки, git-ignored. Остаётся как есть.
- `АГЕНТЫ-ЛОГ.md` — высокоуровневый, сквозной, переживает отдельные фичи и сессии.

Формат одной записи:

```markdown
## 2026-07-19 — [Отдел] — [Краткое название задачи]
- Кто выполнял: я напрямую / суб-агент (implementer/reviewer/...)
- Результат: успех / сбой / эскалация
- Отчёт: ссылка на report-файл, если был
- Примечание: 1-2 предложения по сути, если нужно
```

Правило владения (защита от конфликта одновременной записи, о котором явно предупреждает
лекция): **пишу в этот файл только я**, главный оркестратор, после получения отчёта от
суб-агента или после самостоятельно выполненной работы. Суб-агенты никогда не пишут в него
напрямую.

### 3. Отказоустойчивость (новое)

**Что считается сбоем:** суб-агент вернул статус `BLOCKED`, вернул пустой или бессмысленный
отчёт, либо повторяет одну и ту же ошибку инструмента.

**Протокол:**
1. Сбой логируется в `АГЕНТЫ-ЛОГ.md` (результат: «сбой»).
2. Одна автоматическая повторная попытка — свежий суб-агент с дополненным контекстом (что
   именно пошло не так в первый раз).
3. Если сбой повторился — **не третья попытка автоматически**. Либо я беру задачу на себя
   напрямую (если решение очевидно), либо эскалирую вопрос пользователю в реальном времени
   (если неоднозначность выше моей уверенности).

**Порог:** максимум 1 автоматический ретрай на задачу, не бесконечный.

### 4. Консенсус (новое, частично уже неявно существует)

Таблица приоритета «чьё слово главное» при разногласиях, по убыванию:

1. Явное указание пользователя — всегда выигрывает.
2. Написанный план/спека — выигрывает, **если** ревью не находит в нём конкретного,
   предметного дефекта.
3. Вердикт ревьюера (task-reviewer или финальное ревью) — выигрывает над самооценкой
   исполнителя.
4. Если ревью прямо конфликтует с тем, что явно требует план — не решаю сама, эскалирую
   пользователю с объяснением на понятном языке (уже действующее в этой сессии правило
   «plan-mandated finding arbitration», формализуется здесь как часть общего протокола).

Чек-лист вместо решения «на глаз»: используется уже существующая двух-вердиктная система
ревью (соответствие спеке + качество кода, находки по серьёзности Critical/Important/Minor) —
формализуется как обязательный чек-лист для любого ревью в рамках отдела, а не только для
разработки `kf.py`.

### 5. Чат агентов (переосмыслено под реальную архитектуру инструментов)

В модели Claude Code суб-агенты физически не переписываются друг с другом напрямую — они
отчитываются мне (топология «звезда», не «сетка»). Прямая замена буквального «канала агентов»
из лекции здесь невозможна и не нужна инструментами, которые сейчас используются.

Смысл принципа — избежать бесконечных промежуточных докладов наверх, но сохранить читаемый,
просматриваемый лог операций — закрывается тем же `АГЕНТЫ-ЛОГ.md`: он и есть тот «канал»,
который можно просмотреть как обычную переписку, без необходимости строить настоящую
многоагентную messaging-инфраструктуру.

Если позже будет принято решение о переходе на Вариант Б (`ruflo`), этот принцип можно будет
заменить на настоящий agent-to-agent обмен через его инструменты — не требует пересмотра
остальных пяти принципов.

### 6. Проверка — без изменений, добавлен явный порог

Существующий цикл ревью (`task-reviewer` → фикс → повторное ревью) не меняется. Добавляется
явный потолок, чтобы проверка не превращалась в бесконечный цикл (прямое предупреждение из
лекции): **максимум 2 цикла «найдено → исправлено → повторное ревью» на одну задачу**. Если
после 2 циклов остаются Critical/Important находки — не третья попытка автоматически, а
эскалация пользователю с сутью нерешённой проблемы.

## Эскалация: когда именно

**В реальном времени, в момент возникновения — не собирается в финальный отчёт.** Каждая
находка ревью, каждый конфликт консенсуса, каждый неудачный повторный сбой — сразу к
пользователю (тем же способом, что уже используется всю эту сессию: `AskUserQuestion` с
объяснением на понятном языке).

Эскалация — не «пинговать пользователя на каждый шаг». Обычные успешные шаги тихо
записываются в `АГЕНТЫ-ЛОГ.md`, работа продолжается без остановки. Эскалация происходит
только при пересечении одного из явных порогов:
- принцип 3 (отказоустойчивость): сбой + неудачный единственный ретрай;
- принцип 4 (консенсус): ревью прямо конфликтует с планом;
- принцип 6 (проверка): превышен потолок в 2 цикла ревью.

## Область применения

Протокол применяется **всегда**, независимо от размера задачи (решение пользователя), но
«оркестратор отдела» как отдельный суб-агент дисптачится только для объёмных, действительно
многоагентных задач внутри отдела — не для рутинных однострочных действий (например,
установка одного скилла). Для таких случаев я исполняю роль оркестратора отдела напрямую,
без лишнего слоя диспатча.

## Не входит в эту спеку

- Реальная интеграция `ruflo` (Вариант Б) — отдельная будущая тема, зафиксирована в памяти
  ассистента, не здесь.
- Изменения в уже работающем цикле разработки `kf.py`
  (`brainstorming` → `writing-plans` → `subagent-driven-development`) — не затрагивается.
