# Дизайн: граф знаний (сущности и связи, Kuzu)

Дата: 2026-07-17
Статус: одобрено пользователем, готово к плану реализации

## Контекст

В изначальной архитектуре проекта (`project-config/Архитектура проекта. Настройка и Правила/core.md`)
для LIGHT-профиля был предусмотрен граф знаний на Kuzu (встроенная графовая БД, без JVM,
без отдельного Docker-контейнера) — но эта часть архитектуры так и не была реализована:
текущий `knowledge-factory/kf/` содержит только Qdrant (вектора) + Postgres (метаданные/
дедупликация) + MinIO (сырые файлы) + Redis, без какого-либо графового модуля.

Пользователь попросил реализовать эту функцию. При обсуждении выяснилось, что запрос
можно раскрыть на три разных уровня возможностей:
1. Прямые связи между сущностями ("что связано с X") — граф как структурированный индекс.
2. Multi-hop рассуждение (переход по нескольким связям подряд).
3. Структурированная визуализация/обзор всей базы.

Пользователь выбрал начать с уровня 1 — это техническая основа, на которой уровни 2 и 3
может быть надстроены позже без переделки с нуля (граф данных общий, отличаются только
типы запросов к нему). Уровни 2 и 3 явно вне рамок этой спеки.

## Требования

### Что считается "сущностью"

Широкий набор типов: человек, инструмент, проект, тема, концепт. Тип определяется LLM
при извлечении, не жёстко ограничен списком (см. ниже про парсинг).

### Извлечение — автоматически при `kf.py ingest`

Как и синтез-заметки (`kf/synthesize.py`), извлечение сущностей и связей происходит
автоматически для каждого нового или изменённого файла при каждом запуске `kf.py ingest` —
один дополнительный LLM-вызов на файл (та же модель `LLM_MODEL` через OpenRouter, тот же
уже прочитанный текст файла — не нужен отдельный проход по файлам и не нужна отдельная
команда, которую пользователь мог бы забыть запустить).

### Схема связей — категория + свободное описание

Каждая извлечённая связь между двумя сущностями получает:
- **Категорию** из фиксированного короткого списка: `использует`, `часть_проекта`,
  `связано_с_темой`, `автор_создатель`, `другое`. LLM выбирает одну категорию — это
  надёжная часть извлечения (закрытый список), даёт возможность фильтровать граф по типу
  связи.
- **Свободное текстовое описание** — нюанс связи своими словами (например "использует для
  рендеринга 3D-сцен"), не ограничено списком — сохраняет содержательность, которую жёсткая
  категория одна не передаёт.
- **Путь к файлу-источнику** — из какого файла эта связь извлечена (нужно для
  прослеживаемости и пригодится, когда будет решён отложенный вопрос про очистку графа при
  удалении исходных файлов — см. "Вне рамок").

### Дедупликация сущностей

Одна и та же сущность, упомянутая в разных заметках (например "Blender" в трёх разных
файлах), должна соответствовать **одному** узлу в графе, а не создавать дубликаты.
Сопоставление — по нормализованному имени: обрезка пробелов по краям + приведение к
нижнему регистру для сравнения (оригинальный регистр сохраняется для отображения).
Никакой более сложной семантической дедупликации (fuzzy-matching синонимов вроде
"Blender" и "Блендер") в этой версии не делается — вне рамок.

## Архитектура

- **`kf/graph.py`** (новый модуль, чистые функции, без сети/БД — по образцу `kf/synthesize.py`):
  - `build_extraction_messages(text: str, source_path: str) -> list[dict]` — промпт для LLM,
    просит вернуть JSON со списком сущностей (`name`, `type`) и связей (`from`, `to`,
    `category`, `description`), категория — строго один из пяти значений выше.
  - `extract_entities_and_relationships(settings: Settings, text: str, source_path: str) -> tuple[list[dict], list[dict]]` —
    реальный сетевой вызов (аналог `synthesize_note`), не покрывается автотестом по тому же
    прецеденту проекта, что и другие функции, вызывающие OpenRouter напрямую.
  - `parse_extraction_response(raw_response: str) -> tuple[list[dict], list[dict]]` — парсинг
    JSON-ответа LLM в структуры сущностей/связей. Если `raw_response` не парсится как
    ожидаемый JSON (невалидный JSON, отсутствуют ожидаемые ключи) — функция **поднимает
    `ValueError`**, а не возвращает пустые списки: пустой список сущностей в файле, где они
    реально есть, неотличим от "разбор не удался", поэтому такая двусмысленность недопустима.
    `extract_entities_and_relationships` не перехватывает эту ошибку сама — она всплывает и
    ловится единым `try/except` в `kf/ingest.py`, вместе с ошибками самого сетевого вызова
    (тем же путём, что уже применён к `synthesize_note`).
  - `normalize_entity_name(name: str) -> str` — нормализация для дедупликации (trim +
    нижний регистр).

- **`kf/store/graph_store.py`** (новый модуль, обёртка над Kuzu):
  - `get_connection(settings: Settings) -> kuzu.Connection` — открывает/создаёт файл БД
    Kuzu по пути из настроек.
  - `ensure_schema(conn) -> None` — создаёт таблицы узлов (`Entity`) и связей
    (`RELATED_TO`), если их ещё нет.
  - `upsert_entity(conn, name: str, entity_type: str) -> None` — создаёт узел, если не
    существует (сопоставление по нормализованному имени); не дублирует существующий.
  - `add_relationship(conn, from_name: str, to_name: str, category: str, description: str, source_path: str) -> None` —
    создаёт ребро между двумя уже существующими узлами.
  - `query_entity(conn, name: str) -> list[dict]` — все прямые связи данной сущности
    (с кем/чем связана, категория, описание, файл-источник).

- **Интеграция в `kf/ingest.py`**: после шага синтез-заметки (тем же текстом), вызывается
  извлечение сущностей/связей; при успехе — запись в Kuzu через `graph_store`; при неудаче —
  не прерывает `ingest`, только логирует предупреждение (см. "Обработка ошибок").
  Новые поля `IngestStats`: `entities_extracted: int = 0`, `entities_failed: int = 0`.

- **`kf.py graph <сущность>`** (новая команда в `kf/cli.py`): ищет сущность по
  нормализованному имени, печатает все прямые связи (с кем, категория, описание, файл).

- **`graph_search(entity: str) -> list[dict]`** (новый инструмент в `kf/mcp_server.py` и
  `kf/api.py`): то же самое через MCP — доступно и мне, и Hermes-боту (уже подключён к
  `knowledge-factory` MCP).

## Данные и хранение

Файл базы Kuzu — `knowledge-factory/data/graph/` (путь, уже предусмотренный в исходной
архитектуре под этот профиль, физически на `E:`, как и остальные тома данных). Не в
Docker — Kuzu встроенная библиотека, подключается из процесса `kf.py` напрямую. Модель для
извлечения — существующий `LLM_MODEL` (OpenRouter), отдельная настройка не нужна.

## Обработка ошибок

Сбой извлечения сущностей (ошибка LLM-вызова, некорректный JSON-ответ) не должен прерывать
весь `ingest` — по тому же принципу деградации, что уже применён к синтез-заметкам
(`notes_failed`) и к журналу знаний (после финального фикса в предыдущей фиче). Ошибка
логируется, `entities_failed` увеличивается, обработка остальных файлов продолжается.

## Тестирование

- `kf/graph.py` — чистые юнит-тесты на `build_extraction_messages`, `parse_extraction_response`
  (включая случай некорректного JSON), `normalize_entity_name`. Реальный сетевой вызов
  (`extract_entities_and_relationships`) без автотеста, по прецеденту `kf/synthesize.py`.
- `kf/store/graph_store.py` — тесты на **реальном** Kuzu: временный файл БД через `tmp_path`,
  без Docker (Kuzu — встроенная библиотека) — быстрее и проще существующих Postgres/Qdrant-тестов.
  Проверяют: создание схемы, upsert сущности не создаёт дубликат при повторном вызове с тем
  же (нормализованным) именем, `query_entity` возвращает все прямые связи.
- `kf/ingest.py` — расширение существующих тестов, `extract_entities_and_relationships`
  подменяется моком (как `synthesize_note` уже подменяется).

## Вне рамок (явно)

- **Multi-hop запросы** (переход по нескольким связям подряд) — уровень 2, отложен
  пользователем, будет строиться на той же графовой схеме позже.
- **Визуализация/структурированный обзор графа** — уровень 3, отложен, тоже будет
  использовать те же данные.
- **Очистка графа при удалении исходного файла** — если файл, из которого извлечены
  сущности/связи, исчезает из `raw-data-repository/`, соответствующие рёбра графа сейчас
  не удаляются автоматически. Это прямое продолжение уже отложенного пользователем вопроса
  про очистку Qdrant/Postgres/MinIO при удалении файлов (см. память
  `project_knowledge_journal_deletion_policy.md`) — решается одним и тем же решением
  позже, не дублируется здесь отдельно.
- **Fuzzy-дедупликация сущностей** (синонимы, опечатки, разные языки для одной сущности) —
  только точное совпадение по нормализованному имени в этой версии.
- **Ограничение количества извлекаемых сущностей/связей на файл** (аналог `MAX_VIDEO_FRAMES`
  для видео) — не запрошено пользователем, не добавляется без явного запроса, по прецеденту
  осознанного решения не ограничивать длину LLM-входа для синтез-заметок.
