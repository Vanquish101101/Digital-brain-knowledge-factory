# Digital Brain — правила работы в проекте

Личный «цифровой мозг»: человекочитаемое хранилище знаний (`raw-data-repository/`, в
формате, близком к Obsidian vault) + технический бэкенд семантического поиска
(`knowledge-factory/`, Docker: Qdrant + Postgres + MinIO + Redis, CLI `kf.py`,
Knowledge MCP-сервер).

Полная техническая архитектура, профили развёртывания и список типовых ошибок — в
`project-config/Архитектура проекта. Настройка и Правила/` (`core.md`, `rules.md`,
`ПЕРЕД-СТАРТОМ.md`). Текущий статус и уже принятые решения — в
`project-config/РЕШЕНИЯ-И-СТАТУС.md`. Карта всего проекта — `карта_бз.md` в корне.

## Регламент автодобавления знаний

Когда пользователь присылает новый материал (текст, голосовое, ссылку на видео/статью,
скриншот, PDF, репозиторий, excel-таблицу, фото, изображение и т.п.) с пометкой "добавь в
базу знаний":

1. Определи, к какому разделу `raw-data-repository/` это относится — по таблице
   "Куда класть новое" в `карта_бз.md` (единственный источник правды для маршрутизации):
   сначала проверь совпадение по теме (Маркетинг/3D/Веб-дизайн/Видео/Аудио/Game/хобби —
   тема важнее формата файла), и только если тема не совпадает ни с одним существующим
   разделом — используй общее правило по типу материала.
2. Сохрани/законспектируй материал в этот раздел.
3. Запусти `kf.py ingest`, чтобы материал сразу попал в векторный поиск и в
   `Журнал знаний.md` (журнал пополняется автоматически при индексации, редактировать его
   вручную не нужно).
4. Сообщи пользователю, куда именно добавил (путь к файлу).
5. Обнови `карта_бз.md`, если появился новый файл или папка верхнего уровня.

Ссылки на статьи и YouTube-видео обрабатываются командой `kf.py ingest-url <ссылка> --dest
<раздел>` (агент сам определяет раздел по той же таблице маршрутизации, как и для любого
другого материала, и передаёт его через `--dest`). Если команда предупредила, что извлечённый
текст статьи выглядит скудным — агент вручную достаёт текст через Firecrawl и подменяет им
содержимое сохранённого файла перед повторным `kf.py ingest`.

Если формат всё же непонятен агенту (видеохостинг без субтитров и не YouTube, сайт, который
не читается ни `trafilatura`, ни Firecrawl, и т.п.) — прямо сказать об этом пользователю, а
не притворяться, что материал обработан.

Разовое правило "если/то" при необходимости уточняется отдельно (например, "из тендера
брать только условия") — по умолчанию используется здравый смысл на основе содержания.

## Навигационная карта (карта_бз.md)

`карта_бз.md` в корне проекта — резервный план на случай, если контекст потерян: сначала
читать её, потом продолжать работу. Обновляется при **каждом** добавлении/удалении файлов
и папок верхнего уровня.

## Скиллы и субагенты: локально или глобально

Правило (см. лекцию "Косяк №2"): "Пригодится ли это ещё где-то, кроме этого проекта?"

- **Да, регулярно** → ставить глобально (`~/.claude/skills`).
- **Нет, разовая задача под этот проект** → ставить локально, внутри проекта.

Не ставить всё подряд глобально — один кривой/конфликтующий глобальный скилл ломает
работу во всех проектах одновременно, локальный — только в одном.

## Knowledge Factory (Docker-бэкенд + kf.py + MCP)

- Папка: `knowledge-factory/`. Профиль: **LIGHT** (16GB RAM) — Qdrant +
  Postgres/pgvector + MinIO + Redis, без OpenSearch и Neo4j (слишком прожорливы на JVM).
  Postgres — на именованном томе Docker с фиксированным именем `knowledge-factory_kf_postgres_data`
  (не зависит от имени папки проекта; не на E: — см. РЕШЕНИЯ-И-СТАТУС.md, Windows/NTFS
  bind-mount не даёт нужных Unix-прав).
- LLM для графа/синтеза — **OpenRouter** (через httpx), эмбеддинги — локально (fastembed,
  `paraphrase-multilingual-MiniLM-L12-v2`).
- Тома данных — `knowledge-factory/data/` (папка внутри проекта, физически на
  внешнем диске `E:`, 1.7TB свободно). Настроено в `.env` (DATA_ROOT, MODEL_CACHE_DIR).
  `data/` — в `.gitignore`, большие бинарники (вектора/кэш) в git не идут.
- `OPENROUTER_API_KEY` — отдельный, выпущенный специально под Digital Brain (не общий с
  другими проектами пользователя, по правилу «один проект — один ключ»).
- **Важно:** на `E:\` рядом с папкой `Digital brain` лежит папка «Игры пс4» (не относится к
  проекту). Никогда не сканировать, не индексировать, не трогать и не удалять её или
  что-либо за пределами `E:\Digital brain\`.
- **Проект целиком перенесён с `C:` на `E:` 2026-07-12** (код + vault + данные — всё теперь
  на одном диске). Docker Desktop и Python-рантайм (uv) — по-прежнему на `C:`, это системные
  компоненты, их не переносим никогда.
- **Docker на Hyper-V (не WSL2)** — из-за старой версии Windows. Периодически зависает/роняет
  проброс портов, особенно после перезапуска хоста (проверено на практике). Если контейнер
  долго не отвечает — не пытаться чинить бесконечно, сказать пользователю, чтобы перезапустил
  Docker Desktop вручную; либо `docker restart <контейнер>` часто помогает само по себе.
  Также проверено: `docker compose down` иногда молча ничего не делает (контейнеры остаются
  Up) — в этом случае `docker stop <контейнер>` напрямую работает надёжнее.
- **Переименование папок:** если папка проекта переименовывается, имя Docker Compose проекта
  меняется вместе с ней (берётся из имени текущей папки) — это создаёт **новый** анонимный
  volume вместо переиспользования старого, если у именованных volume'ов нет явного `name:`
  в docker-compose.yml. Все именованные тома здесь имеют фиксированные `name:` — переименование
  папки проекта больше не создаёт пустых дублей.
- `kf.py` (CLI: ingest/search/ask/stats) и Knowledge MCP-сервер (`semantic_search`/`ask`/`stats`,
  зарегистрирован в Claude Code как `knowledge-factory`, scope `user`) — готовы и работают.
- Перед любым `docker compose up` — пройти PRE-FLIGHT из `ПЕРЕД-СТАРТОМ.md` (RAM, диск,
  порты, память Docker).

## Оркестрация работы по отделам

Работа со скиллами, установленными по тематическим «отделам» (веб, продукт, аналитика,
маркетинг — см. `карта_бз.md`/git-историю установки скиллов), следует протоколу из
`docs/superpowers/specs/2026-07-19-department-orchestration-design.md`. Кратко:

- **Отделы:** Веб (`frontend-design`, `web-design-guidelines`, `vercel-react-best-practices`,
  `shadcn`, `site-architecture`, `schema`, `cro`), Продукт (`nodejs-backend-patterns`,
  `fastapi-templates`, `docker-compose-orchestration`, `postgresql-table-design`,
  `qdrant-clients-sdk`, `qdrant-search-quality`, `redis-core`, `onboarding`, `signup`,
  `paywalls`), Аналитика (`analytics`, `ab-testing`), Маркетинг (`copywriting`,
  `copy-editing`, `seo-audit`, `product-marketing`, `customer-research`,
  `marketing-psychology`, `content-strategy`, `ai-seo`, `pricing`, `social`, `emails`,
  `referrals`, `launch`).
- **Роли:** я — главный оркестратор всегда; оркестратор отдела — я же напрямую для рутинных
  задач, отдельный суб-агент-координатор — только для по-настоящему объёмных многоагентных
  задач внутри отдела.
- **Общая память:** каждая операция логируется в `project-config/АГЕНТЫ-ЛОГ.md` (пишу только
  я, суб-агенты не пишут напрямую).
- **Отказоустойчивость:** при сбое суб-агента — один автоматический ретрай со свежим
  контекстом, дальше — беру задачу на себя или эскалирую пользователю, не бесконечный retry.
- **Консенсус:** пользователь > план (если ревью не находит в нём конкретного дефекта) >
  вердикт ревьюера > самооценка исполнителя; конфликт ревью vs план — всегда к пользователю.
- **Проверка:** максимум 2 цикла «найдено → исправлено → ревью» на задачу, дальше —
  эскалация, не бесконечный цикл.
- **Эскалация — в реальном времени**, в момент возникновения (не собирается в финальный
  отчёт), но только при реальном пересечении порога, не на каждый шаг.

Разработка самого `knowledge-factory` (`kf.py`) — отдельный, уже существующий цикл
(`brainstorming` → `writing-plans` → `subagent-driven-development`), этим протоколом не
затрагивается.
