### 1. О чём этот материал

Подробная техническая спецификация и «техническое задание» на проектирование и развёртывание автономной базы знаний («Цифрового мозга»). Описывается архитектура системы, стек технологий, жизненный цикл обработки данных и процесс инициализации проекта, а также решаются вопросы постоянного хранения информации и однократного подключения внешних источников.

### 2. Ключевые идеи и факты

- **Главная задача:** Создать модульную, self-hosted фабрику знаний, которая автономно перерабатывает любые входные данные (текст, изображения, видео) в структурированное хранилище (векторная БД, графовая БД, память агентов) без необходимости повторного подключения исходного носителя.
- **Архитектура и стек:**
    - **Ядро:** MinIO (S3-хранилище файлов), QDrant (векторная БД), PostgreSQL и Redis.
    - **Расширенный слой:** Neo4j/Kuzu (графовая БД), MemO/Letta (память), OpenSearch, LlamaGraph, LangGraph, LiteLLM.
    - **Метод поиска:** Hybrid RAG + Graph RAG + reranking + RBAC-aware retrieval.
    - **Связующий слой:** Knowledge MCP — единый протокол, через который все агенты (в Docker) обращаются к единой базе знаний.
- **Процесс обработки (конвейер):**
    1. **Raw (Ingest):** Приём исходных файлов.
    2. **Extraction:** Извлечение текста и таблиц.
    3. **Chunking:** Нарезка на смысловые куски.
    4. **Embedding:** Векторизация кусков.
    5. **Vector Index:** Индексация и поиск по смыслу.
    6. **Graph Builder:** Построение графа связей между сущностями.
    7. **Memory Store:** Сохранение контекста агента.
    8. **Retrieval, Ranking, Synthesis, Agent Router:** Поиск, сортировка, сборка ответа и маршрутизация.
- **Правила работы с данными:**
    - **Размещение томов:** Все тома данных (MinIO, Qdrant, Postgres, Neo4j и т.д.) обязательно должны находиться на отдельном или внешнем диске (bind-mount), чтобы не забить системный диск.
    - **Постоянство памяти:** После однократного анализа внешнего диска данные сохраняются в постоянное хранилище (векторы, графы, файлы), и система может работать с ними без повторного подключения источника.
- **Принципы системы:** AI-First, Retrieval-First, Agent-First, Memory-First, Self-hosted, Docker, K8s-Ready.
- **Процесс запуска:** `docker compose up` → индексация (Scan + extract) → Вектор (embeddings ready) → Граф (relations linked) → Память (context warm) → Система онлайн.

### 3. Как это может пригодиться в будущих задачах и проектах

- **Готовая дорожная карта для быстрого старта:** Материал содержит чёткую последовательность действий (прочитать core.md → rules.md → изучить ошибки → проверить структуру папок). Её можно использовать как чек-лист при развёртывании любой self-hosted AI-системы.
- **Шаблон архитектуры:** Представленный стек (MinIO + QDrant + Neo4j + Knowledge MCP + Docker) является готовым референсным чертежом для создания хранилища знаний, которое само объясняет агентам, как с ним работать.
- **Правила для инфраструктуры:** Чёткое указание на размещение данных на отдельном диске и использование bind-mount — это практическая инструкция, предотвращающая типичные проблемы с нехваткой места на системном диске.
- **Проектирование асинхронного рабочего процесса:** Описанный сценарий («подключил внешний диск → проанализировал → отключил → система помнит навсегда») можно применить для организации безопасной работы с конфиденциальными данными, которые не должны храниться на постоянном подключении.
- **Фабрика знаний как ядро для будущих проектов:** Принцип формирования единого «ядра» базы знаний на основе анализа данных пользователя (характер, мышление, проекты) закладывает фундамент для создания персонального AI-ассистента, который понимает контекст без повторного обучения.
