# Digital Brain — правила работы в проекте

Личный «цифровой мозг»: человекочитаемое хранилище знаний (`Хранилище входных данных/`, в
формате, близком к Obsidian vault) + технический бэкенд семантического поиска
(`Цифровой мозг (digital-brain)/`, Docker: Qdrant + Postgres + MinIO + Redis, CLI `kf.py`,
Knowledge MCP-сервер).

Полная техническая архитектура, профили развёртывания и список типовых ошибок — в
`Файлы настройки проекта/Архитектура проекта. Настройка и Правила/` (`core.md`, `rules.md`,
`ПЕРЕД-СТАРТОМ.md`). Текущий статус и уже принятые решения — в
`Файлы настройки проекта/РЕШЕНИЯ-И-СТАТУС.md`. Карта всего проекта — `карта_бз.md` в корне.

## Регламент автодобавления знаний

Когда пользователь присылает новый материал (текст, голосовое, ссылку на видео/статью,
скриншот, PDF, репозиторий и т.п.) с пометкой "добавь в базу знаний":

1. Определи, к какому разделу `Хранилище входных данных/` это относится (Входящие, если не
   очевидно сразу — Знания, Ресурсы, Проекты, Задачи/Цели и т.д.).
2. Сохрани/законспектируй материал в этот раздел.
3. Сообщи пользователю, куда именно добавил (путь к файлу).
4. Обнови `карта_бз.md`, если появился новый файл или папка.

Если формат непонятен агенту (например, чтение конкретных ссылок/аудио ещё не
настроено) — прямо сказать об этом, а не притворяться, что материал обработан.

Разовое правило "если/то" при необходимости уточняется отдельно (например, "из тендера
брать только условия") — по умолчанию используется здравый смысл на основе содержания.

## Навигационная карта (карта_бз.md)

`карта_бз.md` в корне проекта — резервный план на случай, если контекст потерян: сначала
читать её, потом продолжать работу. Обновляется при **каждом** добавлении/удалении файлов
и папок верхнего уровня.

## Скиллы и субагенты: локально или глобально

Правило (см. лекцию "Косяк №2"): "Пригодится ли это ещё где-то, кроме этого проекта?"

- **Да, регулярно** → ставить глобально (`~/.Codex/skills`).
- **Нет, разовая задача под этот проект** → ставить локально, внутри проекта.

Не ставить всё подряд глобально — один кривой/конфликтующий глобальный скилл ломает
работу во всех проектах одновременно, локальный — только в одном.

## Knowledge Factory (Docker-бэкенд + kf.py + MCP)

- Папка: `Цифровой мозг (digital-brain)/`. Профиль: **LIGHT** (16GB RAM) — Qdrant +
  Postgres/pgvector + MinIO + Redis, без OpenSearch и Neo4j (слишком прожорливы на JVM).
  Postgres — на именованном томе Docker с фиксированным именем `knowledge-factory_kf_postgres_data`
  (не зависит от имени папки проекта; не на E: — см. РЕШЕНИЯ-И-СТАТУС.md, Windows/NTFS
  bind-mount не даёт нужных Unix-прав).
- LLM для графа/синтеза — **OpenRouter** (через httpx), эмбеддинги — локально (fastembed,
  `paraphrase-multilingual-MiniLM-L12-v2`).
- Тома данных — `Цифровой мозг (digital-brain)/data/` (папка внутри проекта, физически на
  внешнем диске `E:`, 1.7TB свободно). Настроено в `.env` (DATA_ROOT, MODEL_CACHE_DIR).
  `data/` — в `.gitignore`, большие бинарники (вектора/кэш) в git не идут.
- `OPENROUTER_API_KEY` — отдельный, выпущенный специально под Digital Brain (не общий с
  другими проектами пользователя, по правилу «один проект — один ключ»).
- **Важно:** на `E:\` рядом с папкой `Digital brain` лежит папка «Игры пс4» (не относится к
  проекту). Никогда не сканировать, не индексировать, не трогать и не удалять её или
  что-либо за пределами `E:\Digital brain\`.
- **Проект целиком перенесён с `C:` на `E:` 2026-07-12** (код + vault + данные — всё теперь
  на одном диске). Docker Desktop и Python-рантайм (uv) — по-прежнему на `C:`, это системные
  компоненты, их не переносим никогда.
- **Docker на Hyper-V (не WSL2)** — из-за старой версии Windows. Периодически зависает/роняет
  проброс портов, особенно после перезапуска хоста (проверено на практике). Если контейнер
  долго не отвечает — не пытаться чинить бесконечно, сказать пользователю, чтобы перезапустил
  Docker Desktop вручную; либо `docker restart <контейнер>` часто помогает само по себе.
  Также проверено: `docker compose down` иногда молча ничего не делает (контейнеры остаются
  Up) — в этом случае `docker stop <контейнер>` напрямую работает надёжнее.
- **Переименование папок:** если папка проекта переименовывается, имя Docker Compose проекта
  меняется вместе с ней (берётся из имени текущей папки) — это создаёт **новый** анонимный
  volume вместо переиспользования старого, если у именованных volume'ов нет явного `name:`
  в docker-compose.yml. Все именованные тома здесь имеют фиксированные `name:` — переименование
  папки проекта больше не создаёт пустых дублей.
- `kf.py` (CLI: ingest/search/ask/stats) и Knowledge MCP-сервер (`semantic_search`/`ask`/`stats`,
  зарегистрирован в Codex как `knowledge-factory`, scope `user`) — готовы и работают.
- Перед любым `docker compose up` — пройти PRE-FLIGHT из `ПЕРЕД-СТАРТОМ.md` (RAM, диск,
  порты, память Docker).

## Отчёт по итогам задачи

После заметных изменений — коротко резюмировать: что сделано, что добавилось/поменялось,
что осталось сделать пользователю руками (например, "подключи диск и пришли путь").
