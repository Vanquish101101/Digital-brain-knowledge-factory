# Текущая архитектура Knowledge Factory (актуально)

> Эта заметка — действующая версия архитектуры базы знаний Digital Brain. Более старая
> заметка `010 Архив (Неиспользуемое)/Процесс работы.txt` описывает раннюю концепцию проекта
> (Neo4j, OpenSearch, MemO/Letta) — эти компоненты не используются, заметка оставлена в
> архиве как исторический черновик, а не как текущее описание системы.

## Стек (профиль LIGHT, 16GB RAM)

- **Векторная БД:** Qdrant (не Pinecone, не облачный сервис — локальный Docker-контейнер).
- **Реляционная БД:** PostgreSQL + pgvector.
- **Файловое хранилище:** MinIO (S3-совместимое).
- **Кэш/очереди:** Redis.
- **Граф знаний:** Kuzu (embedded, без отдельного сервера) — не Neo4j.
- **Без:** OpenSearch, Neo4j, MemO/Letta — сознательно исключены из LIGHT-профиля (слишком
  прожорливы на JVM/RAM для этого масштаба).
- **Эмбеддинги:** локально, fastembed, модель `paraphrase-multilingual-MiniLM-L12-v2`.
  Есть переключаемые облачные профили через OpenRouter (`qwen3-8b`, `openai-small`,
  `openai-large`), но по умолчанию — локальная модель.
- **LLM для графа/синтеза заметок:** OpenRouter (через httpx).

## Хранилище знаний (vault)

Человекочитаемые файлы физически лежат на диске: `E:\Digital brain\raw-data-repository\`
(Obsidian-подобная структура: `001 Входящие`, `002 Мышление`, `003 Знания`, `004 Инструменты`,
`005 Ресурсы`, `006 Задачи`, `007 Проекты`, `008 Интересы`, `010 Архив`,
`Синтезированные данные (synthesized-notes)`). Это не Notion и не облачный сервис — прямой
доступ к содержимому есть только у процессов с доступом к этой файловой системе.

## CLI и MCP

- `kf.py` (в `knowledge-factory/`) — команды: `ingest`, `ingest-url`, `sync-deletions`,
  `search`, `ask`, `stats`, `graph`, `embedding-model list/use/sync`.
- MCP-сервер `knowledge-factory` — **4 инструмента**: `semantic_search`, `ask`, `stats`,
  `graph_search`. Никаких инструментов загрузки файлов через MCP нет — файлы добавляются
  напрямую в vault на диске, затем индексируются командой `kf.py ingest`.

## Пайплайн индексации (файл → векторы), кратко

1. Сканирование `raw-data-repository/` (`ingest_directory`), фильтр по scope.
2. Дедуп по SHA256 контента (`needs_ingest`) — не переиндексирует неизменённые файлы.
3. Извлечение текста по формату (`extract_text`): текст/PDF/docx/xlsx, изображение —
   OCR + vision-caption, аудио/видео — транскрипция Whisper (+ разбор кадров для видео).
4. Чанкинг по абзацам (`chunk_text`, ~1500 символов, overlap 150).
5. Эмбеддинг чанков (`embed_for_profile`) → запись в Qdrant + MinIO + Postgres.
6. Синтез заметки через LLM (`synthesize_note`) — сама переиндексируется как отдельный документ
   в `Синтезированные данные (synthesized-notes)/`.
7. Извлечение сущностей и связей (`extract_entities_and_relationships`) → граф в Kuzu.
8. Запись в `Журнал знаний.md`.

Полная схема с диаграммой — `project-config/Архитектура проекта. Настройка и Правила/pipeline.md`
(этот файл вне vault, не индексируется семантическим поиском — техническая документация
проекта, не часть базы знаний).

## Обратный путь: удаление

`kf.py sync-deletions` — сравнивает vault с тем, что проиндексировано, находит реально
удалённые файлы (отличая от переименований по хэшу), чистит Postgres, все коллекции Qdrant,
MinIO и граф знаний. Обычный `ingest` не удаляет ничего сам.
